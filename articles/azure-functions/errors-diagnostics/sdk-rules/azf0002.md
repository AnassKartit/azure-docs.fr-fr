# <a name="azf0002-inefficient-httpclient-usage"></a>AZF0002 : utilisation inefficace de HttpClient

Cette règle est déclenchée quand un [HttpClient](/dotnet/api/system.net.http.httpclient?view=netcore-3.1&preserve-view=true) est instancié dans une méthode décorée `[FunctionName]`.

| | Value |
|-|-|
| **Identificateur de la règle** |AZF0002|
| **Catégorie** |[Fiabilité]|
| **Gravité** |Avertissement|

## <a name="rule-description"></a>Description de la règle

Utilisation simple de [`HttpClient`](/dotnet/api/system.net.http.httpclient?view=netcore-3.1&preserve-view=true) pour que les requêtes HTTP présentent plusieurs problèmes, notamment la vulnérabilité à l’épuisement des sockets. Dans une application de fonction, l’appel du constructeur `HttpClient` dans le corps d’une méthode de fonction créera une nouvelle instance avec chaque appel de fonction, en amplifiant ces problèmes. Pour les applications qui s’exécutent sur un [plan d’hébergement de consommation](../../consumption-plan.md), une utilisation `HttpClient` inefficace peut épuiser les [limites de connexion sortantes](/azure/azure-resource-manager/management/azure-subscription-service-limits.md#azure-functions-limits)du plan.

La [meilleure pratique recommandée](/dotnet/architecture/microservices/implement-resilient-applications/use-httpclientfactory-to-implement-resilient-http-requests.md) consiste à utiliser un [`IHttpClientFactory`] avec une injection de dépendances _ou_ une instance `HttpClient` statique unique, en fonction de la nature de votre application.

## <a name="how-to-fix-violations"></a>Comment corriger les violations

+ Déplacez votre instanciation `HttpClient` vers une instance statique définie en dehors de la fonction, comme dans [cet exemple](../../manage-connections.md?tabs=csharp#http-requests).

+ Fournissez un `HttpClient` à vos fonctions à l’aide de [IHttpClientFactory](/dotnet/api/system.net.http.ihttpclientfactory?view=dotnet-plat-ext-5.0&preserve-view=true) via l’[injection de dépendances](../../functions-dotnet-dependency-injection.md). C’est l’approche la plus recommandée pour résoudre cette violation.

## <a name="when-to-suppress-the-rule"></a>Quand supprimer la règle

Cette règle ne doit pas être supprimée.
